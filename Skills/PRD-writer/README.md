# PRD-writer 使用说明

`PRD-writer` 用于把原始需求材料分阶段整理成需求分析文档、Plan 方案、原型和标准 PRD，适合需要先确认需求理解，再确认方案与页面交互，最后输出正式产品需求文档的场景。

## 什么时候用

- 你有口头描述、聊天记录、截图或零散需求，想先整理成需求分析文档
- 你已经有一份需求分析文档，想校正后继续生成方案、原型和 PRD
- 你想先确认产品方案，再确认页面原型，最后生成 PRD
- 你要补充已有 PRD 的某些章节
- 你要按统一模板输出产品需求文档

## 怎么用

最推荐的方式是直接给我原始需求材料或 `.md` 需求分析文档，然后说：

- `根据这些需求先整理需求分析文档，再出 Plan、原型和 PRD`
- `根据这个需求分析文档先校正需求分析，再出 Plan、原型和 PRD`
- `帮我补充这个 PRD 的数据字典`
- `把这个需求整理成标准 PRD`

## 你需要提供什么

- 原始需求材料，或已有需求分析文档
- 明确的业务目标
- 目标系统边界
- 现有约束和已知规则

## 我会输出什么

我会按四个阶段输出：

1. **需求分析文档**：按照 `reference/requirement-analysis-template.md` 输出背景、目标、问题、范围、业务场景、业务规则和待确认项，其中业务场景必须逐条罗列
2. **Plan 方案**：需求分析确认后，按照 `reference/plan-template.md` 输出范围、流程、规则、使用者与系统交互场景、UC 用例、系统边界和原型建议，其中交互场景和 UC 都是必填内容
3. **原型**：Plan 确认后，按照 `reference/prototype-template.md` 输出页面、字段、按钮、状态和交互说明
4. **PRD**：原型确认后，按照 `reference/prd-template.md` 的 10 段式结构输出正式 PRD

PRD 的 10 段式结构包括：

1. 功能定义
2. 业务规则
3. 流程说明
4. 数据来源
5. 页面交互
6. 操作后发生什么
7. 约束与限制
8. 状态机
9. 用例
10. 数据字典

## 写作前检查

如果需求涉及库存变动、库存占用/释放、扣减/回补、调拨或出入库，开始写 PRD 之前必须先确认：

- 目标系统是 `OMS`、`WMS` 还是 `TMS`
- 库存责任归属由哪个系统承担

## 重要规则

- 不能只靠口头描述直接写 PRD，必须先整理成需求分析文档
- 需求分析阶段必须罗列业务场景清单
- 不能跳过需求分析确认直接生成 Plan
- Plan 阶段必须确认使用者与系统的交互场景
- Plan 阶段必须输出 UC 用例清单
- 不能跳过 Plan 确认直接生成原型
- 不能跳过原型确认直接生成 PRD
- 未明确的信息要标注为 `待确认`
- 涉及流程分支时，必须使用 Mermaid
- 用户要求“重写”或“新建”时，要输出到新的 `.md` 文件，不覆盖原文件

## 需求分析文档模板

如果你还没有需求分析文档，可以先提供口头描述、聊天记录或零散材料，`PRD-writer` 会先按 `reference/requirement-analysis-template.md` 整理成标准需求分析文档，并等待你确认。模板末尾附了一个“到货签收”的填写示例，方便你直接照着写。
