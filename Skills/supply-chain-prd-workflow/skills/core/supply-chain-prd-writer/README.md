# 供应链 PRD Writer 使用说明

`supply-chain-prd-writer` 用于把原始需求材料分阶段整理成需求分析文档、Plan 方案、原型和标准 PRD，目标是让普通供应链业务人员也能按专业的供应链软件产品总监标准输出可直接协同研发的产出物。它适合需要先确认需求理解，再确认方案与页面交互，最后输出正式产品需求文档的场景。

## 套件关系

这套能力按“主入口 + 辅助 skill”来设计，不把所有职责塞进一个 skill。

- `supply-chain-prd-writer`：主入口，负责供应链后台文档的四阶段编排
- `Problem Statement`：先把问题、阻塞点和场景说清楚
- `Prd Development` / `Create Prd`：生成或补全通用 PRD
- `Strategy Red Team`：定稿前挑逻辑漏洞
- `Pre Mortem`：上线前预演风险
- `Release Notes`：对外输出版本说明

如果你当前要做的是供应链后台需求文档，优先从这个主入口开始；如果你只是做问题定义、风险审查或发版说明，就直接用更专门的 skill。

## 什么时候用

- 你有口头描述、聊天记录、截图或零散需求，想先整理成需求分析文档
- 你已经有一份需求分析文档，想校正后继续生成方案、原型和 PRD
- 你想先确认产品方案，再确认页面原型，最后生成 PRD
- 你要补充已有 PRD 的某些章节
- 你要按统一模板输出产品需求文档

## 怎么用

最推荐的方式是直接给我原始需求材料或 `.md` 需求分析文档，然后说：

- `根据这些需求先整理需求分析文档，再出 Plan、原型和 PRD`
- `根据这个需求分析文档先校正需求分析，再出 Plan、原型和 PRD`
- `帮我补充这个 PRD 的数据字典`
- `把这个需求整理成标准 PRD`

## 你需要提供什么

- 原始需求材料，或已有需求分析文档
- 明确的业务目标
- 目标系统边界
- 现有约束和已知规则

## 我会输出什么

我会按四个阶段输出：

1. **需求分析文档**：按照 `reference/requirement-analysis-template.md` 输出背景、目标、问题、范围、业务场景、业务规则和待确认项，其中业务场景必须逐条罗列
2. **Plan 方案**：需求分析确认后，按照 `reference/plan-template.md` 输出范围、流程、规则、使用者与系统交互场景、UC 用例、系统边界和原型建议，其中交互场景和 UC 都是必填内容；对于供应链、WMS、TMS 场景，还会补足能直接给研发评审、拆任务和联调的细化内容
3. **原型**：Plan 确认后，按照 `reference/prototype-template.md` 先输出 Markdown 多页静态稿，确认后如需再转 HTML 页面稿；原型会尽量写成研发和测试可以直接拿去对齐的页面执行稿，而不是只有外观示意。若用户明确要求，也可以直接切换为 HTML、可交互或图片原型
4. **PRD**：原型确认后，按照 `reference/prd-template.md` 的 10 段式结构输出正式 PRD

## 原型阶段定义

原型阶段的最终产出物是面向后台管理工作台的 Markdown 多页静态页面稿。它按页面清单、布局、字段、按钮、状态、异常和页面流转，把主流程、分支和边界讲清楚，不要求可点击模拟；用户确认后，如需更贴近实现，再补 HTML 页面稿。原型只负责页面怎么用、怎么变、怎么流转，不混入需求分析、方案、接口或数据字典。

需求分析文档、Plan 文档和 PRD 文档都必须包含文档头信息：

- 标题
- 文档类型：需求 / Plan / PRD
- 版本
- 日期
- 作者
- 相关方

PRD 的 10 段式结构包括：

1. 功能定义
2. 业务规则
3. 流程说明
4. 数据来源
5. 页面交互
6. 操作后发生什么
7. 约束与限制
8. 状态机
9. 用例
10. 数据字典

## 写作前检查

如果需求涉及库存变动、库存占用/释放、扣减/回补、调拨或出入库，开始写 PRD 之前必须先确认：

- 目标系统是 `OMS`、`WMS` 还是 `TMS`
- 库存责任归属由哪个系统承担

## 重要规则

- 不能只靠口头描述直接写 PRD，必须先整理成需求分析文档
- 需求分析阶段必须罗列业务场景清单
- 不能跳过需求分析确认直接生成 Plan
- Plan 阶段必须确认使用者与系统的交互场景
- Plan 阶段必须输出 UC 用例清单
- 不能跳过 Plan 确认直接生成原型
- 不能跳过原型确认直接生成 PRD
- 需求分析文档、Plan 文档、PRD 文档都必须包含标题、文档类型、版本、日期、作者、相关方
- 未明确的信息要标注为 `待确认`
- 涉及流程分支时，必须使用 Mermaid
- 用户要求“重写”或“新建”时，要输出到新的对应格式文件，不覆盖原文件；需求分析、Plan、PRD 默认输出 `.md`，原型默认输出 `.md` 的多页静态稿，确认后可再输出 `.html`

## 需求分析文档模板

如果你还没有需求分析文档，可以先提供口头描述、聊天记录或零散材料，`supply-chain-prd-writer` 会先按 `reference/requirement-analysis-template.md` 整理成标准需求分析文档，并等待你确认。模板末尾附了一个“到货签收”的填写示例，方便你直接照着写。
