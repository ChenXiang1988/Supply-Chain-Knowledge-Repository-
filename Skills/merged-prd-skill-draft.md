# 合并版 PRD 技能草案

## 结论

可以合并，但不建议把 `create-prd` 和 `prd-writer-v2` 直接拼接成一个超长模板。

更稳妥的做法是做成一个“主技能 + 双模式”：

- **模式 A：策略型 PRD**，吸收 `create-prd` 的通用起草能力
- **模式 B：结构化 PRD**，吸收 `prd-writer-v2` 的后台/业务系统写法

这样可以保留两者优势，同时避免模板冲突和输出冗余。

---

## 推荐定位

**建议技能名**：`prd-writer`

**建议描述**：
从模糊想法或需求分析文档生成 PRD；支持策略型 PRD 和后台/业务系统 PRD，自动标记待确认项，并输出可评审、可落地的文档。

**建议类型**：`workflow`

**核心目标**：

- 让用户只说一句话也能起草 PRD
- 让已有需求分析文档能快速整理成规范 PRD
- 对后台、审批流、状态机、权限、数据字典等内容保持高完整度

---

## 合并思路

### 保留 `create-prd` 的部分

- 低门槛输入
- 先问关键问题，再写文档
- Summary / Objective / Segment / Value Proposition / Release 这些策略型章节
- 面向更广泛协作人群的通俗表达

### 保留 `prd-writer-v2` 的部分

- 事实优先，不猜
- `待确认` 明确标注
- Mermaid 流程图和状态机
- 业务规则、页面交互、系统影响、验收标准、数据字典
- 后台/业务系统优先

### 不建议直接合并的部分

- 8 段式和 10 段式主结构不要同时并列
- 不要同时把“市场分段/价值主张”和“数据字典/状态机”当成同等强制项
- 不要让一个技能既像战略文档生成器，又像后台 PRD 规格化工具，却没有路由逻辑

---

## 推荐的技能结构

### 1. 入口判断

先判断输入属于哪一类：

| 输入类型 | 处理方式 |
|---|---|
| 一句话想法、功能点、问题描述 | 走策略型 PRD 模式 |
| 需求分析文档、后台流程、审批流、工单、风控、配置类需求 | 走结构化 PRD 模式 |
| 混合输入 | 先抽取事实，再按结构化 PRD 输出 |

### 2. 策略型 PRD 模式

适合早期、信息不完整、需要先对齐方向的场景。

建议输出结构：

1. Summary
2. Background
3. Objective
4. Target Users / Segments
5. Value Proposition
6. Solution Overview
7. Open Questions
8. Release / Phasing

### 3. 结构化 PRD 模式

适合已经有分析材料、要进入评审或开发前定稿的场景。

建议输出结构：

1. 功能定义
2. 业务规则
3. 流程说明
4. 数据来源
5. 页面交互
6. 操作后发生什么
7. 约束与限制
8. 状态机
9. 用例与验收标准
10. 数据字典

---

## 合并后的硬规则

1. 不确定的内容统一标记 `待确认`
2. 条件分支、异常分支、状态流转必须用 Mermaid
3. 后台功能优先补齐权限、批量、导入导出、日志、通知、异步处理
4. 任何关键需求都要能落到验收标准
5. 不要因为输入模糊就编造细节
6. 输出要根据输入成熟度自动切换模板

---

## 推荐的冲突处理

### 如果只给一句话需求

优先追问 1 个关键问题，或者先按策略型 PRD 输出初稿。

### 如果给了需求分析文档

优先走结构化 PRD 模式，不要退回到通用模板。

### 如果既要战略表达，又要后台细节

把前 2 到 3 章用于策略背景，把后面章节用于结构化细节。

---

## 你最终会得到什么

- 一个技能就能覆盖“从想法到 PRD”的全链路
- 早期产品讨论和后期评审落地都能用
- 不会丢掉 `create-prd` 的通用性
- 也不会丢掉 `prd-writer-v2` 的严谨性
