# 3PL 平台系统约束清单

文档类型：系统约束  
版本：V0.1  
日期：2026-06-18  
作者：Codex  
相关方：产品、架构、研发、仓储运营、关务、财务、客户成功

## 1. 目标

这份文档用来明确平台在设计和落地时必须遵守的边界。

如果约束不先说清楚，后面很容易变成“功能都要、责任都不清、上线后也不好复制”。

## 2. 约束总表

| 约束类别 | 约束内容 | 实现建议 | 风险 |
|---|---|---|---|
| 全局核心 | 一个全球核心，不按国家拆主系统 | 核心流程统一，差异用配置承载 | 变成多套系统 |
| 仓型配置 | 国内仓、海外仓、保税仓只能作为配置维度 | 仓型、监管属性、运营模式独立配置 | 代码分叉失控 |
| 库存真相 | 库存以流水为准，快照只做查询加速 | 建立库存流水账和库存快照双层模型 | 库存口径混乱 |
| 订单与履约 | 订单和履约要分离 | 一个订单可拆成多个任务和发运单 | 业务链路耦合 |
| 执行与结算 | 执行层和计费层要分层 | 作业事件与计费规则分开存储 | 账务和作业互相污染 |
| 对外协同 | 客户和合作方不能只靠人工传文件 | Portal + API + 文档中心 | 协同效率低 |
| 权限 | 外部角色最小权限，内部操作可审计 | RBAC + 数据范围 + 审计日志 | 数据外泄或误操作 |
| 合规 | 保税、清关、放行、单证必须可追溯 | 监管对象独立建模，状态留痕 | 无法满足审计 |
| 集成 | 外部接口必须可重试、可幂等 | 标准化事件和回传协议 | 接口对不齐 |
| 扩张 | 首批必须可复制到更多国家 / 仓型 | 把国家规则和仓型规则配置化 | 无法规模化 |

## 3. 系统边界建议

当前这套平台更适合定义为“3PL 平台核心中台”，而不是单一 WMS。

### 3.1 平台应该负责的内容

- 组织与权限
- 仓网与仓型配置
- 订单接入与路由
- 库存流水与快照
- 仓内任务编排
- 客户协同和合作方协同
- 计费与对账
- 关务资料和状态
- 经营看板与异常控制塔

### 3.2 平台不应该一开始就承担的内容

- 所有国家的本地税务细节
- 所有物流商的深度私有协议
- 所有报表的全量即时报送
- 所有客户的个性化流程

## 4. 首批试点边界建议

首批试点建议固定成“一小块可控区域”：

- 一个国家
- 一种仓型
- 一个客户
- 一条主流程
- 一条核心接口链路

如果首批边界都不收敛，后续文档会全部失去约束力。

## 5. 待确认项

| 待确认项 | 为什么重要 |
|---|---|
| 首批是国内仓、海外仓还是保税仓 | 会影响合规、流程和字段设计 |
| 首批是否需要客户门户 | 决定协同层是否前置 |
| 首批是否需要合作方回传 | 决定接口和权限模型 |
| 首批是否涉及计费闭环 | 决定是否要把财务一起纳入试点 |
