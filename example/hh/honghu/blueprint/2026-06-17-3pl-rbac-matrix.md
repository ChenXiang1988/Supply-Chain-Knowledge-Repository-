# 3PL 平台角色权限矩阵

状态：草案  
日期：2026-06-17  
关联蓝图：`docs/superpowers/specs/2026-06-17-global-3pl-platform-blueprint-design.md`

## 1. 目标

这份文档定义全球 3PL 平台的角色权限模型，覆盖内部运营团队、客户、海外合作方三类使用者。

目标不是“谁能点哪个按钮”，而是：

- 谁能看什么数据
- 谁能在哪个范围内操作
- 谁能审批
- 谁能导出
- 谁能配置

## 2. 权限原则

### 2.1 先按组织隔离，再按仓库隔离，再按客户隔离

默认的数据边界顺序建议如下：

1. 租户 / 组织
2. 区域
3. 仓库
4. 客户
5. 订单 / SKU / 任务级别

### 2.2 外部角色最小权限

客户和合作方只应看到自己的数据和被授权的动作，不应默认看到平台全局信息。

### 2.3 审批与执行分离

可以操作的不一定可以审批；可以审批的不一定可以配置。

### 2.4 高风险动作必须留痕

涉及以下内容的操作必须审计：

- 库存调整
- 关务放行相关变更
- 费率修改
- 对账核销
- 权限变更

## 3. 权限动作定义

为便于统一，权限动作分为：

- `R` = 读取
- `W` = 新增 / 修改
- `A` = 审批 / 确认
- `X` = 导出 / 下载
- `C` = 配置 / 管理
- `D` = 删除 / 作废

## 4. 角色定义

### 4.1 内部角色

- 总部运营
- 区域负责人
- 仓库经理
- 仓库作业员（试点范围已细化为 收货员/验货员/上架员/拣货员/复核员/打包员/发运员/叉车工/库存管理员，见用户画像V2）
- 合规 / 关务
- 财务
- 客户经理 / CS
- 系统管理员

### 4.2 外部角色

- 客户 / 货主
- 海外合作方 / 本地执行伙伴
- 第三方承运商
- 外部系统服务账号

## 6. 数据范围规则

### 6.1 总部运营

- 可看全局仓网
- 可看全局经营指标
- 不直接做具体仓内操作

### 6.2 区域负责人

- 可看本区域和下属仓
- 可管理区域内合作方
- 不应越区访问其他区域客户数据

### 6.3 仓库经理

- 可管理本仓作业
- 可查看本仓库存和异常
- 可审批本仓作业例外

### 6.4 仓库作业员

- 只看被分派的任务和必要作业信息
- 不看客户全量订单和财务信息

### 6.5 合规 / 关务

- 只看与申报、单证、放行相关的数据
- 可发起合规拦截

### 6.6 财务

- 只看与合同、账单、应收应付相关的数据
- 不直接改写仓内执行事实

### 6.7 客户 / 货主

- 只能看自己组织下的数据
- 可下单、查库存、看状态、下载单证、确认账单

### 6.8 海外合作方

- 只能看分配给自己的仓和任务
- 只能处理被授权的操作回传和状态确认

## 7. 特殊控制点

### 7.1 库存调整

必须经过审批或双人复核，且保留调整原因、前后值和操作者。

### 7.2 保税和关务操作

申报、放行、监管库存变更必须有单证链和审计链。

### 7.3 费率变更

客户费率、仓储费、附加费和罚扣规则必须有版本和生效时间。

### 7.4 数据导出

导出必须区分：

- 全量导出
- 当前筛选导出
- 脱敏导出

### 7.5 外部接口账号

服务账号建议采用：

- 机器身份认证
- 最小权限
- 接口级限流
- 调用审计

## 8. 建议的权限模型

建议使用“角色 + 数据范围 + 动作权限 + 审计规则”的四层模型：

1. 角色定义职责
2. 数据范围定义能看哪里
3. 动作权限定义能做什么
4. 审计规则定义高风险动作怎么留痕

这样可以避免把权限做成大量硬编码特例。
