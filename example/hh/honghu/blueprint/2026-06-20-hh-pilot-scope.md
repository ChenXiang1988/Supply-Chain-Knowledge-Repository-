# 泓湖 3PL 平台试点范围

状态：草案  
日期：2026-06-20  
作者：Martin  
相关方：产品、架构、研发、仓储运营

## 1. 背景

蓝图（[全球3PL平台蓝图设计](./2026-06-17-global-3pl-platform-blueprint-design.md)）已经冻结，场景清单、用户画像、系统约束、领域模型等基础材料已完成。下一步不是继续产文档，而是把第一批要做的事情圈定下来——确定一个可验证的闭环范围，让团队知道"第一版到底做什么、不做什么"。

## 2. 核心决策

| 维度 | 决策 | 决策人 | 理由 |
|---|---|---|---|
| 首批仓型 | 国内仓 | — | 规则最简单，无清关复杂度，能最快跑通闭环；海外仓作为第二期 |
| 首批客户类型 | B2B + B2C 混合 | — | 验证系统同时支持批量出库和单件零售的能力 |
| 客户门户 | 首批包含 | — | 协同层是产品能力而非附属功能（蓝图原则4.6），首批就必须验证客户自助下单和查库存 |
| 计费闭环 | 暂不纳入第一期 | — | 先验证执行链，计费与对账后置 |
| 合作方回传 | 不涉及 | — | 国内仓无海外合作方 |
| 保税/清关 | 不涉及 | — | 国内仓无清关场景 |

## 3. 首批场景

从 [场景清单](../../../scenarios/2026-06-18-3pl-scenario-list.md) 12个场景中，依据上述决策做裁剪。

### 3.1 进入第一批（7个）

| 编号 | 场景名称 | 进入理由 |
|---|---|---|
| S-01 | 客户接入与账号开通 | 混合客户意味着 B2B 和 B2C 两种身份都要能开账号 |
| S-02 | 主数据初始化 | SKU、仓库配置、服务规则是全部后续场景的前置条件 |
| S-03 | 入库预约与到货登记 | 国内物流链路入口，没有预约后面收货无依据 |
| S-04 | 收货、验货、上架 | 仓内执行第一步，库存入账的起点 |
| S-05 | 库存查询与占用 | 国内仓库存可视化，也是客户门户的核心能力——"我的库存在哪里" |
| S-06 | 出库下单与承诺 | 客户侧最关键动作，B2B和B2C要有区别但共用核心逻辑 |
| S-07 | 拣货、复核、打包、发运 | 仓内执行后半段，与S-04构成完整的仓内正向闭环 |

### 3.2 暂不纳入第一批（5个）

| 编号 | 场景名称 | 裁剪理由 |
|---|---|---|
| S-08 | 退货入库与处理 | P1，先跑通正向链路，逆向链路后置 |
| S-09 | 保税 / 清关单证处理 | 国内仓不具备此场景的前提条件 |
| S-10 | 异常工单与处置 | P0 但在第一期可先走线下记录流程，系统化第二期补 |
| S-11 | 计费与对账 | 决策暂不纳入第一期 |
| S-12 | 经营看板与 SLA 监控 | P1，先跑通执行链，再看经营指标 |

## 4. 首批用户角色

依据 [用户画像](../../../scenarios/2026-06-18-3pl-user-personas.md) ，7个场景覆盖的角色范围：

| 角色 | 使用终端 | 本期主要动作 | 数据权限范围 |
|---|---|---|---|
| 仓库经理 | Web + Handheld | 查看作业面板、人员排班、处理异常 | 所管仓库 |
| 班组长（入库） | Handheld + Web | 分派收货/验货/上架单、确认差异、上报异常 | 本班组任务 |
| 班组长（出库） | Handheld + Web | 分派拣货/复核/打包/发运任务、查看完成进度 | 本班组任务 |
| 收货员 | Handheld | 到货登记、扫描确认收货 | 所领任务 |
| 验货员 | Handheld | 逐箱扫描验货、差异登记、拍照 | 所领任务 |
| 上架员 | Handheld | 扫描 SKU、确认库位、上架入库 | 所领任务 |
| 拣货员 | Handheld | 领取任务、扫描库位+SKU、拣货执行 | 所领任务 |
| 复核员 | Handheld | 逐件扫描复核、确认差异 | 所领任务 |
| 打包员 | Handheld + 标签打印机 | 确认打包清单、打印标签/面单 | 所领任务 |
| 发运员 | Handheld | 发运确认 | 所领任务 |
| 叉车工 | Handheld | 楼层转运、大件/整托搬运、补货移库 | 所领转运任务 |
| 客户管理员 | Portal | SKU 注册与贴码管理、提交预约/订单、查库存和轨迹 | 仅限本组织数据 |
| 总部运营 | Web | 看全局订单和库存、配置规则、定义库区类型模板 | 全量 |
| 客户成功 | Web | 客户入驻引导、查看客户订单/库存/异常、处理客户问询 | 所负责客户 |

不纳入第一期的角色：区域负责人（等海外仓）、合规/关务（国内仓无需求）、财务（计费后置）、海外合作方（国内仓无需求）。

## 5. 系统边界

### 5.1 第一期做

| 层 | 能力范围 | 关键说明 |
|---|---|---|
| 主数据与权限 | 组织、仓库、SKU 主数据、客户档案、角色与 RBAC | 支撑多客户（B2B + B2C）同时在线 |
| 订单与承诺 | 订单接入、校验、库存承诺、合单/拆单 | B2B 订单支持批量行项，B2C 支持单品下单 |
| 库存中心 | 库存流水、库存快照、库存查询 | 流水为事实来源，快照提速 |
| 仓储执行 | 收货 → 上架 → 拣货 → 复核 → 打包 → 发运 | 支持 Handheld 扫描操作 |
| 协同层 | 客户门户（下单 + 库存查询 + 订单轨迹） | 客户可自助操作，不依赖邮件和 Excel |

### 5.2 第一期不做

| 层 | 能力范围 | 原因 |
|---|---|---|
| 关务与合规 | 保税、清关、申报、监管库存 | 国内仓无此需求 |
| 计费与结算 | 费率卡、仓储计费、操作计费、对账、开票 | 决策后置 |
| 经营看板 | 实时看板、SLA 监控、异常控制塔 | P1，先跑通执行链 |
| 合作方门户 | 接单、回传、对账 | 海外仓才需要 |
| 退货 | 退货登记、质检、入库/报废 | P1 |
| 跨仓转运 | 调拨单、中间运输回传 | 国内单仓模式下不涉及 |
| 海外/保税仓复制 | 多国多仓型复制 | 第二期 |
| 外部电商平台对接 | 淘宝/京东/Amazon/Shopee等 | 第一期以人工下单替代 |

## 6. 核心链路

第一期需要验证的一条完整链路：

```
客户接入(S-01) → 主数据初始化(S-02) → 入库预约(S-03) → 收货验货上架(S-04)
                                                              ↓
拣货复核打包发运(S-07) ← 出库下单与承诺(S-06) ←── 库存可视(S-05)
        ↓
    发运回传(状态变更)
        ↓
   客户门户看到轨迹
```

### 6.1 B2B 链路特征

- 订单行项多，单个订单可能包含多种 SKU
- 支持整箱出库和拆零出库
- 拣货可合并波次

### 6.2 B2C 链路特征

- 订单行项少，通常单件单品
- 出库频次高，单量可能远大于 B2B
- 对发运时效敏感

### 6.3 混合客户的关键验证点

同一个仓库里，B2B 和 B2C 两种订单模式共用同一套库存、同一套执行流程，但：
- 下单页面不同（批量 vs 单件）
- 拣货策略可能不同（波次 vs 逐单）
- 打包规格不同（整箱 vs 快递包裹）

## 8. 从"待确认"到"已确认"

对照 [用户画像](../../../scenarios/2026-06-18-3pl-user-personas.md) 和 [系统约束](../../../scenarios/2026-06-18-3pl-system-constraints.md) 中的待确认项：

| 原待确认项 | 本文档确认结果 |
|---|---|
| 首批试点是国内仓、海外仓还是保税仓 | **国内仓** |
| 首批客户是 B2B、B2C 还是混合 | **B2B + B2C 混合** |
| 首批是否需要客户门户 | **需要** |
| 首批是否需要合作方回传 | **不需要**（国内仓无合作方） |
| 首批是否涉及计费闭环 | **不涉及** |

## 9. 下一步

试点范围冻结后，建议立即进入 P0 场景需求展开。建议优先展开以下 3 个场景（构成最短闭环）：

1. **S-04 收货验货上架** — 仓内执行起点
2. **S-06 出库下单与承诺** — 客户侧核心动作
3. **S-07 拣货复核打包发运** — 仓内执行终点
