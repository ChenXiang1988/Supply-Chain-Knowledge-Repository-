## 2. 场景描述

客户完成主数据初始化（S-02）后，需要把即将到达仓库的货物提前通知仓库，让仓库有准备时间。仓库收到预约后安排收货窗口，并在车辆实际到达时做"到货登记"，把入库订单从"已预约"推进到"已到货"，触发 S-04 作业流程。

## 3. 场景主干

```
客户提交预约 → 仓库确认预约 → 车辆实际到达 → 到货登记 → 触发 S-04
```

## 4. 子场景展开

### 4.1 子场景 A：客户提交入库预约（客户 → 仓库）

| 字段 | 内容 |
|---|---|
| 角色 | 客户（门户操作） |
| 触发条件 | 客户已备好货，安排物流送往仓库 |
| 前置条件 | 客户已完成 S-02 主数据初始化（SKU、仓库已配置） |

**正常流程：**
1. 客户登录门户，进入"入库预约"页面
2. 选择目标仓库
3. 填写预约信息：

| 信息项 | 说明 | 本期必填 |
|---|---|---|
| 预计到仓日期 | 单选 | 是 |
| 预计到仓时段 | 上午/下午/晚上，支持多选 | 是 |
| 运输方式 | 整车 / 零担 / 快递 / 自送 | 是 |
| 车辆信息 | 车牌号（如已知） | 否 |
| 司机/联系人 | 姓名 + 电话 | 否 |
| 货物清单 | SKU + 预报数量 + 箱数（明细表） | 是 |

4. 货物清单填写方式（本期）：

| 方式 | 适用场景 | 操作 |
|---|---|---|
| 手动逐行录入 | SKU 数量少 | 添加 SKU → 填预报数量 → 填箱数 |
| 模板导入 | SKU 数量多 | 下载 Excel 模板 → 填好上传 → 系统解析 |
| 复制历史预约 | 同类到货频繁 | 从历史预约复制清单再微调 |

5. 客户提交预约
6. 系统生成入库订单，状态为"待确认"

**B2B 与 B2C 差异：**

| 维度 | B2B 客户 | B2C 客户 |
|---|---|---|
| 货物清单规模 | 常见 10-50 行 | 常见 1-5 行 |
| 到货频率 | 低频大批量 | 高频小批量 |
| 运输方式 | 整车/零担为主 | 快递/自送为主 |
| 推荐填写方式 | 模板导入 | 手动录入或复制历史 |

**边界情况：**
- 同一入库订单内 SKU 不能重复
- 预报数量必须 > 0
- 到仓日期不能是过去日期
- 系统需校验所选仓库在客户的可操作范围内

### 4.2 子场景 B：仓库确认预约（仓库 → 系统）

| 字段 | 内容 |
|---|---|
| 角色 | 仓库经理 |
| 触发条件 | 客户已提交预约，状态为"待确认" |
| 终端 | Web |

**正常流程：**
1. 仓库经理在 Web 端查看"待确认"预约列表（按到仓日期排序）
2. 点击入库订单查看详情

**仓库经理需要确认的信息：**

| 确认项 | 说明 |
|---|---|
| 货物清单 | SKU 是否都在主数据中、预报量是否合理 |
| 到仓时段 | 当日该时段是否有余量（人工判断） |
| 特殊要求 | 客户备注（如"需要低温存储"、"卸货需叉车"） |

3. 仓库经理可执行以下操作：

| 操作 | 结果 |
|---|---|
| 确认 | 预约状态变为"已确认"，生成到货签收单 |
| 调整时段 | 给客户建议新的到仓时段，预约回到"待客户确认" |
| 拒绝 | 注明原因，预约关闭 |

4. 确认后，系统的"收货计划看板"中新增一条记录

**本期处理：**
- 时段容量不做系统级校验，仓库经理人工判断
- 预约确认后不给客户发通知（第二期补自动通知）

### 4.3 子场景 C：到货登记（收货员）

| 字段 | 内容 |
|---|---|
| 角色 | 收货员 |
| 触发条件 | 运输车辆实际到达仓库 |
| 终端 | Handheld |

**正常流程：**
1. 车辆到达，司机/送货人出示入库订单号或运单号
2. 收货员在 Handheld 上打开"到货登记"
3. 输入或扫描入库订单号
4. 系统拉取该预约的基本信息（客户、SKU 清单、预报箱数）
5. 作业员核对实际到货的外部包装情况：

| 核对项 | 动作 |
|---|---|
| 车辆信息 | 核对车牌（如有预约填写） |
| 箱数/托盘数 | 实际清点，录入系统 |
| 外包装状态 | 目测：正常 / 有破损 / 有变形 |

6. 作业员点击"确认到货"
7. 入库订单状态从"已确认"变为"已到货"
8. 系统自动触发 S-04 作业流程——生成收货任务，收货区待办列表中新增一条

**边界情况：**

| 情况 | 处理方式 |
|---|---|
| 无未预约的车辆到达（临时送货） | 作业员联系仓库经理，由经理在 Web 端紧急补建入库订单，再推进到货登记 |
| 实际到货与预约严重不符（如少了几托） | 作业员在"到货登记"时备注差异，仍确认到货，差异在 S-04 验货阶段详细登记 |
| 车辆严重迟到（超过预约日） | 入库订单状态保留为"已确认"，车辆到达时仍可做登记，但系统标记"超时到货"供后续分析 |

## 5. 入库订单状态流转

```mermaid
stateDiagram-v2
    [*] --> 待确认 : 客户提交
    待确认 --> 已确认 : 仓库经理确认
    待确认 --> 已拒绝 : 仓库经理拒绝
    待确认 --> 待客户确认 : 仓库经理建议调整时段
    待客户确认 --> 待确认 : 客户接受新时段
    待客户确认 --> 已取消 : 客户取消
    已确认 --> 已到货 : 到货登记
    已到货 --> S-04收货中 : 触发作业流程
    S-04收货中 --> 已收货 : S-04完成
    已拒绝 --> [*]
    已取消 --> [*]
```

## 6. 与前后场景的关联

### 前置依赖 S-02
- 客户必须已完成主数据初始化
- 目标仓库必须在客户的授权范围内
- SKU 必须在主数据中存在

### 后置支撑 S-04
- S-04 作业流程的起点是"已到货"状态的入库订单
- ASN 明细（即预约中的货物清单）是 S-04 验货比对的基准
- 到货登记时记录的实际箱数，是 S-04 收货员第一手参考
