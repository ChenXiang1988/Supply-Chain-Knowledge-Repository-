# S-04 收货、验货、上架

文档类型：场景需求

相关方：产品、架构、研发、仓储运营

## 1. 场景定位

| 项目 | 内容 |
|---|---|
| 场景编号 | S-04 |
| 场景名称 | 收货、验货、上架 |
| 所属链路 | 入库主链路 |
| 前置场景 | S-03 入库预约与到货登记 |
| 后置场景 | S-05 库存查询与占用 |
| 优先级 | P0 |

## 2. 场景描述

货物到达仓库后，由入库班组（收货员、验货员、上架员）在系统引导下依次完成收货确认、验货核对、差异登记和上架入库。整个过程从"货物在码头"变成"库存可用"，是仓内执行链的起点。

## 3. 用户与角色

| 角色 | 动作 | 终端 |
|---|---|---|
| 收货员 | 扫描入库订单号、清点确认到货 | Handheld |
| 验货员 | 逐箱扫描验货、差异登记、拍照 | Handheld |
| 上架员 | 扫描 SKU、确认库位、上架入库 | Handheld |
| 班组长（入库） | 分派任务、确认差异处理方案、审批拒收 | Handheld + Web |
| 仓库经理 | 查看收货进度、处理重大异常 | Web |

## 4. 主流程

```
货物到仓 → 到货签收 → 验货核对 → 差异处理 → 上架分配 → [楼层转运] → 上架确认 → 库存入账
```

### 4.1 第一步：到货签收

| 字段 | 内容 |
|---|---|
| 触发条件 | 车辆到达仓库、货物卸至收货区 |
| 操作角色 | 收货员 |
| 前置数据 | 入库入库订单（来自 S-03）、ASN 预报 |

**正常流程：**
1. 收货员扫描入库订单号或选择未完成的入库订单
2. 系统展示该预约对应的 ASN 明细（SKU、预报数量、箱数）
3. 收货员确认实际到货的箱数、托盘数，提交到货签收
4. 系统创建到货签收单，状态为"已收货待验货"

**本期简化：**
- 第一期不做 ASN 系统对接，客户通过门户提交入库预约时手动填写货物清单
- 收货员手持终端扫描入库订单上的条形码即可拉取清单

### 4.2 第二步：验货核对

| 字段 | 内容 |
|---|---|
| 操作角色 | 验货员 |
| 操作位置 | 收货区 / 验货区 |

**正常流程：**
1. 验货员逐箱或逐托扫描 SKU 条码
2. 系统自动比对 SKU 与 ASN 是否匹配
3. 验货员核对数量、外观完整性
4. 对于 B2C 订单相关的到货（如新品入库），检查效期（按 S-02 注册时声明的保质期推算到期日）
5. 验货通过，系统记录"验货合格"
6. 验货合格的商品进入"待上架"状态

**B2B 与 B2C 差异：**

| 维度 | B2B 到货 | B2C 到货 |
|---|---|---|
| 包装单位 | 整箱/整托 | 散件或小箱 |
| 验货精度 | 抽检为主 | 全检或高比例抽检 |
| 效期要求 | 看行业 | B2C 必查效期 |
| 条码来源 | 厂家箱码 | 可能含商品69码 |

**本期处理：**
- 第一期统一按"逐箱扫描 + 抽检"流程
- 效期检查为必做项：客户在 S-02 注册 SKU 时已声明有效期管理方式，验货员需核对并录入生产日期，系统自动推算到期日

### 4.3 第三步：差异处理（异常分支）

在验货过程中出现的差异，按以下分支处理：

**情况一：数量差异（实收 < 预报）**

| 字段 | 内容 |
|---|---|
| 处理人 | 验货员（登记）→ 班组长（确认） |
| 系统动作 | 1. 作业员标记差异 SKU 和少收数量<br>2. 系统生成验货入库单（差异行项）<br>3. 班组长确认后，按实收数量继续入库<br>4. 验货入库单（差异行项）推送至客户门户（本期不做自动推送，第二期补） |
| 库存结果 | 按实收数量入账，预报与实收差异留痕 |

**情况二：数量差异（实收 > 预报）**

| 字段 | 内容 |
|---|---|
| 处理人 | 验货员（登记）→ 班组长（确认） |
| 系统动作 | 1. 作业员登记多收的 SKU 和数量<br>2. 班组长确认后，可选择：拒绝多收部分（退还）或全部接收<br>3. 接收部分正常入库 |
| 库存结果 | 按确认后的数量入账 |

**情况三：外包装破损**

| 字段 | 内容 |
|---|---|
| 处理人 | 验货员（拍照登记）→ 班组长（判断） |
| 系统动作 | 1. 作业员拍照并标记破损<br>2. 系统创建破损记录，关联到到货签收单<br>3. 班组长判断：轻微破损（正常入库）还是严重破损（隔离待处理） |
| 库存结果 | 严重破损的商品进入"不合格品区"，不入可用库存 |

**情况四：SKU 不匹配（错货）**

| 字段 | 内容 |
|---|---|
| 处理人 | 验货员（标记）→ 班组长（确认拒收） |
| 系统动作 | 1. 扫描后系统提示 SKU 不在 ASN 中<br>2. 作业员标记为"错货"<br>3. 整箱/整托隔离，不入库 |
| 库存结果 | 错货不入库，退回给货主 |

### 4.4 第四步：上架分配

| 字段 | 内容 |
|---|---|
| 操作角色 | 系统自动 + 仓库经理（可选干预） |

**系统推荐规则：**
1. 根据 SKU 属性（尺寸、重量、周转率）匹配库位类型
2. 优先推荐已有同 SKU 的库位
3. 平层仓：优先推荐靠近收货区的空库位
4. 楼栋仓：按 SKU 流速度推荐楼层（快流→3F 拣货区，中慢流→4-9F 存储区）
5. 重货推荐低位库位/低楼层，轻货推荐高位库位/高楼层

**本期简化：**
- 第一期上架规则采用固定策略（同 SKU 优先合并、新 SKU 就近），不做周转率分析
- 仓库经理可以在 Web 端手动调整上架库位
- Handheld 端显示"推荐库位"（含楼层信息），上架员可以接受或手动选择其他库位

### 4.5 第五步：楼层转运（楼栋仓专有）

| 字段 | 内容 |
|---|---|
| 触发条件 | 当前货物所在楼层 ≠ 推荐库位所在楼层 |
| 操作角色 | 叉车工 |
| 终端 | Handheld |

**楼层转运判断逻辑：**

```
当前楼层 == 目标楼层？
  ├── 是 → 跳过转运，进入 4.6 上架确认（平层仓永远走此分支）
  └── 否 → 生成楼层转运单
              ↓
           叉车工 Handheld 领取任务
              ↓
           叉车工运送货物至目标楼层
              ↓
           扫描目标楼层确认到位
              ↓
           上架员收到通知 → 进入 4.6
```

**本期处理：**
- 楼层转运单由叉车工手动领取（不做自动派单）
- 转运过程中该 SKU 在 Handheld 上架页暂时隐藏，转运完成后再显示
- 转运中的货物状态 = "楼层转运中"

### 4.6 第六步：上架确认

| 字段 | 内容 |
|---|---|
| 操作角色 | 上架员 |
| 操作终端 | Handheld |

**正常流程：**
1. 上架员扫描 SKU 条码
2. 系统显示推荐库位（或上架员手动录入库位）
3. 上架员将商品放入库位，扫描库位条码确认
4. 系统校验 SKU 与库位是否匹配（如库位有品类限制）
5. 确认后，库存从"待上架"变为"可用"
6. 库存流水记录一条"上架入库"记录

**批量上架（B2B 整托场景）：**
1. 上架员扫描托盘码
2. 系统自动关联该托盘上的所有 SKU
3. 一次性确认整个托盘上架到指定库位

## 5. 场景结束判定

收货验货上架场景结束的标志：

| 判定条件 | 系统状态 |
|---|---|
| 该到货签收单内所有 SKU 验货完成且差异已确认 | 到货签收单状态 = "已收货" |
| 所有"待上架"商品已完成上架确认 | SKU 库存状态 = "可用" |
| 破损/错货已生成对应的差异工单 | 异常商品状态 = "隔离" |

## 6. 与前后场景的关联

### 前置依赖 S-03

- 必须有已生成的入库入库订单（含 ASN 明细）
- 入库订单上的预报数量是验货比对的基准

### 后置支撑 S-05

- 上架确认后，库存流水和库存快照同步更新
- 客户在门户查库存时，应能看到"新入库"的库存变化

## 8. 本期试点边界

| 本期做 | 本期不做 |
|---|---|
| 基于入库订单的到货签收 | ASN 系统对接 |
| 逐箱扫描验货 + 外观检查 | 质检规则引擎 |
| 四项差异处理（少收/多收/破损/错货） | 自动通知客户差异 |
| 固定策略库位推荐（含楼层维度） | 周转率驱动的智能上架 |
| 楼栋仓楼层转运单（叉车工） | 货梯调度优化 |
| 扫描确认上架 | 语音拣货/上架 |
| 验货入库单（差异行项）留存 | 差异自动触发工单（S-10 第一期不做） |
