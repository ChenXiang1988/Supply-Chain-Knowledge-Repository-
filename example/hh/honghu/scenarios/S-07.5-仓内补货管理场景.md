## 1. 场景概述

### 1.1 场景编号
**S-07.5** - 仓内补货管理（Replenishment Management）

### 1.2 场景名称
仓内补货管理

### 1.3 业务背景
在"存拣分离"模式下，仓库分为**存储区**（高密度存放）和**拣货区**（快速拣选）。当拣货区库存不足时，需要从存储区补充货物，这个过程称为"补货"。

补货管理的核心目标：
- 确保拣货区不断货，保障订单履约效率
- 优化补货时机和数量，降低库存持有成本
- 提供灵活的补货策略，适应不同SKU特性
- 实现补货任务的全链路可视化和闭环管理

### 1.4 适用仓型
| 仓型 | 补货特征 | 说明 |
|---|---|---|
| **楼栋仓**（多楼层） | 跨楼层补货 | 存储区在高层（如6F），拣货区在低层（如3F），需乘电梯 |
| **平层仓** | 同层补货 | 存储区和拣货区在同一平面，通过通道连接 |
| **自动化仓** | AGV/堆垛机补货 | 系统自动调度设备执行补货 |

## 2. 补货策略模型

### 2.1 四种经典补货模型

系统设计应支持以下四种补货模型，用户可根据SKU特性灵活配置。

#### **模型A：(s, S) 最小-最大模型**

**原理：**
- 当库存 ≤ s（最低警戒线）时触发补货
- 补货到 S（最高目标库存）

**适用场景：**
- 缺货后果严重的关键物料
- 需求波动大、不可预测的商品
- 示例：生产线关键备件、医疗急救耗材

**配置参数：**
```
s = 再订货点（Reorder Point, ROP）
S = 目标库存水平（Target Stock Level）

计算逻辑：
- 触发条件：库存 ≤ s
- 补货数量 = S - 当前库存
```

**示例：**
```
s = 50件（触发点）
S = 200件（目标库存）
当前库存 = 45件（≤ 50，触发补货）
补货数量 = 200 - 45 = 155件
```

#### **模型B：(s, Q) 固定批量模型**

**原理：**
- 当库存 ≤ s（触发点）时触发补货
- 每次补固定数量 Q

**适用场景：**
- 快消品、标准包装工业品
- 按箱/托补货的场景
- 示例：饮料、标准件、整箱商品

**配置参数：**
```
s = 触发点（Trigger Point）
Q = 固定补货量（Fixed Reorder Quantity）

计算逻辑：
- 触发条件：库存 ≤ s
- 补货数量 = Q（固定值）
```

**示例：**
```
s = 20件
Q = 100件（固定补货量）
当前库存 = 18件（≤ 20，触发补货）
补货数量 = 100件
```

#### **模型C：(R, S) 定期检查模型**

**原理：**
- 按固定时间间隔 R 检查库存
- 补货到目标库存 S（补货量可变）

**适用场景：**
- SKU多、单品价值不高
- 缺货影响不大的商品
- 示例：便利店文具、办公耗材、附件类商品

**配置参数：**
```
R = 检查周期（Review Cycle），如每天、每周
S = 目标库存水平

计算逻辑：
- 触发条件：到达检查时间点
- 补货数量 = S - 当前库存
```

**示例：**
```
R = 每天检查（每日补货）
S = 100件
检查日当前库存 = 30件
补货数量 = 100 - 30 = 70件
```

#### **模型D：(S-1, S) 出一补一模型**

**原理：**
- 每卖出/拣出一件，立即补一件
- 维持库存水平 S 不变

**适用场景：**
- 价值极高、不可或缺、不允许囤货的物料
- 示例：高端设备零件、医疗植入物、贵重样品

**配置参数：**
```
S = 目标库存水平（通常较小）

计算逻辑：
- 触发条件：每次拣货完成后
- 补货数量 = 拣货数量（出一补一）
```

**示例：**
```
S = 5件（始终保持5件）
拣货1件 → 立即补货1件 → 库存恢复5件
```

### 2.2 补货模型选择决策树

```
SKU特性分析
├── 价值极高、不允许缺货？
│   └── 是 → 模型D：(S-1, S) 出一补一
│
├── 需求波动大、不可预测？
│   └── 是 → 模型A：(s, S) 最小-最大
│
├── 按固定包装补货（箱/托）？
│   └── 是 → 模型B：(s, Q) 固定批量
│
└── 需求稳定、可预测？
    └── 是 → 模型C：(R, S) 定期检查
```

## 3. 补货触发机制

### 3.1 触发方式分类

| 触发方式 | 说明 | 适用场景 |
|---|---|---|
| **库存阈值触发** | 库存降至预警值时自动触发 | 模型A、模型B |
| **定时触发** | 按固定时间周期触发 | 模型C |
| **订单驱动触发** | 接收订单后实时检查并触发 | 紧急订单、模型D |
| **人工手动触发** | 仓库人员手动创建补货任务 | 特殊情况、紧急补货 |

### 3.2 触发判断逻辑

#### **关键决策：使用哪种库存作为判断依据？**

| 判断依据 | 问题 | 推荐 |
|---|---|---|
| **可用库存**（Available Stock） | 库位上库存尚未被拣走，补货会导致容量不够；配货单多时会不断触发补货 | ❌ 不推荐 |
| **账面库存**（Book Stock） | 更准确反映实际库存状态，包含已分配但未拣货的数量 | ✅ 推荐 |
| **物理库存**（Physical Stock） | 实际盘点数量，可能包含损坏、锁定库存 | ⚠️ 辅助参考 |

**推荐逻辑：**
```
触发条件判断使用：账面库存（Book Stock）
账面库存 = 物理库存 - 锁定库存 - 已分配库存

补货数量计算使用：可用库存（Available Stock）
可用库存 = 物理库存 - 锁定库存
```

### 3.3 触发时机配置

在WMS系统中，每个SKU可配置：

| 参数 | 说明 | 示例值 |
|---|---|---|
| **补货触发值**（Reorder Point） | 库存降至此值时触发补货 | 50件 |
| **目标库存值**（Target Stock） | 补货到这个目标值 | 200件 |
| **最小补货量**（Min Replen Qty） | 建议是包装值的倍数 | 10件（1箱） |
| **最大补货量**（Max Replen Qty） | 库位能容纳的最大值 - 触发值 | 150件 |
| **检查周期**（Review Cycle） | 定期检查的时间间隔 | 每天 / 每周 |
| **安全库存**（Safety Stock） | 应对需求波动的缓冲库存 | 30件 |

## 4. 补货任务管理

### 4.1 补货任务生成

#### **生成方式**

1. **自动生成**（推荐）
   - 系统后台定时任务（如每5分钟）扫描拣货区库存
   - 根据补货策略自动生成补货任务
   - 生成后进入"待领取"状态

2. **手动生成**
   - 仓库人员在PC端或Handheld端手动创建补货任务
   - 适用于：紧急补货、特殊处理

#### **补货任务单据结构**

参考 `3PL平台单据结构` 文档（第3.9节），补货单（Replenishment Task）字段：

| 字段 | 说明 | 来源 | 必填 |
|---|---|---|---|
| `replen_id` | 补货任务唯一编号 | 系统自动生成（如：REPL-20260620-001） | ✅ |
| `replen_type` | 补货类型 | 系统判断（自动/手动） | ✅ |
| `priority` | 优先级 | 系统计算（高/中/低） | ✅ |
| `pick_task_id` | 关联拣货任务ID | 订单驱动补货时关联 | ❌ |
| `sku_id` | SKU ID | 系统检测 | ✅ |
| `sku_name` | SKU名称 | 主数据 | ✅ |
| `sku_barcode` | SKU条码 | 主数据 | ✅ |
| `pack_spec` | 包装规格 | 主数据（如：1箱=10件） | ✅ |
| `qty_required` | 需补货数量 | 系统计算 | ✅ |
| `qty_actual` | 实际补货数量 | 执行时填写 | ❌ |
| `from_zone` | 来源区域 | 系统查找（如：6F存储区） | ✅ |
| `from_location` | 来源库位 | 系统查找（如：6F-C-05-02） | ✅ |
| `to_zone` | 目标区域 | 系统确定（如：3F拣货区） | ✅ |
| `to_location` | 目标库位 | 系统确定（如：3F-A-01-03） | ✅ |
| `container_code` | 中转容器编号 | 系统分配或人工指定 | ❌ |
| `assigned_to` | 分配给（叉车工ID） | 系统分配或人工指定 | ❌ |
| `status` | 任务状态 | 系统管理 | ✅ |
| `created_at` | 创建时间 | 系统自动记录 | ✅ |
| `completed_at` | 完成时间 | 系统自动记录 | ❌ |
| `created_by` | 创建人 | 系统记录或人工填写 | ✅ |
| `remarks` | 备注 | 人工填写 | ❌ |

#### **补货任务状态机**

```
待领取（Pending）
    ↓ 叉车工领取
补货中（In Progress）
    ↓ 完成确认
已到位（Completed）
    ↓ 取消
已取消（Cancelled）
```

### 4.2 补货任务分配

#### **分配策略**

1. **自动分配**（推荐）
   - 系统根据叉车工位置、工作量、技能等级自动分配
   - 算法：最近空闲人员优先

2. **手动分配**
   - 仓库经理在PC端手动指定叉车工
   - 适用于：特殊任务、培训场景

3. **抢单模式**
   - 叉车工在Handheld端主动领取补货任务
   - 适用于：人员充足、任务简单的场景

## 5. 补货执行流程

### 5.1 整体流程图

```
[库存监控] → [触发补货] → [生成补货任务] → [任务分配] 
    ↓
[叉车工领取任务] → [前往存储区取货] → [扫描确认取货] 
    ↓
[运送到拣货区] → [扫描确认放货] → [补货完成] → [库存更新]
```

### 5.2 PC端补货流程

**适用场景：** 楼栋仓、批量补货、培训场景

**操作步骤：**

1. **查看补货任务列表**
   - 路径：仓储管理 → 补货管理 → 补货任务列表
   - 筛选条件：任务状态、SKU、库位、创建时间

2. **选择补货任务**
   - 勾选"待领取"状态的补货任务
   - 支持批量选择（如一次处理10个任务）

3. **执行补货**
   - 点击"完成"按钮
   - 系统自动执行库存转移：
     - 存储区库存减少
     - 拣货区库存增加
     - 生成库存流水记录

4. **打印补货清单**（可选）
   - 点击"打印"按钮
   - 生成补货任务清单（包含SKU、数量、库位信息）
   - 叉车工按清单执行

**优点：**
- 操作简单，适合新手
- 支持批量处理
- 有完整的操作记录

**缺点：**
- 需要回到PC端操作，效率较低
- 不适合跨楼层、大仓库场景

### 5.3 Handheld端补货流程（推荐）

**适用场景：** 所有仓型，特别是楼栋仓

**操作步骤：**

#### **步骤1：领取补货任务**

```
Handheld端操作流程：
1. 登录Handheld → 选择"补货管理"
2. 查看"待领取"任务列表
3. 选择任务（或"一键领取所有任务"）
4. 任务状态变为"补货中"
```

#### **步骤2：补货移出（从存储区取货）**

```
操作流程：
1. 查看任务详情：SKU、数量、来源库位
2. 前往来源库位（如：6F-C-05-02）
3. 扫描库位条码 → 确认库位正确
4. 扫描SKU条码 → 确认SKU正确
5. 输入取货数量 → 确认
6. 系统提示："已将X件移至中转容器Y"
7. 将货物放入中转容器（如：周转箱、托盘）
```

**关键设计：**
- **中转容器管理**
  - 领取任务时，系统自动分配中转容器编号
  - 或：叉车工扫描空容器条码进行绑定
  - 容器类型：周转箱（小件）、托盘（大件）、笼车（中件）

#### **步骤3：运送货物**

```
楼栋仓场景：
1. 乘坐电梯从6F到3F
2. 到达3F拣货区
3. 扫描目标库位条码

平层仓场景：
1. 驾驶叉车/推车前往拣货区
2. 到达目标库位
3. 扫描目标库位条码
```

#### **步骤4：补货移入（放到拣货区）**

```
操作流程：
1. 扫描目标库位条码 → 确认库位正确
2. 扫描SKU条码 → 确认SKU正确
3. 扫描中转容器条码 → 确认货物正确
4. 输入放货数量 → 确认
5. 系统提示："补货完成！"
6. 任务状态变为"已到位"
```

#### **步骤5：确认完成**

```
系统自动执行：
1. 更新库存：
   - 存储区库存减少
   - 拣货区库存增加
2. 生成库存流水：
   - 流水类型：补货移库（Replenishment Transfer）
   - 来源库位：6F-C-05-02
   - 目标库位：3F-A-01-03
   - 数量：X件
   - 操作人：叉车工ID
   - 操作时间：完成时间
3. 通知拣货员：
   - Handheld推送消息："3F-A-01-03库位已补货，可继续拣货"
   - 或：在拣货任务列表中高亮显示
```

### 5.4 补货路径优化

**问题：** 多个SKU需补货时，如何规划最优路线？

**解决方案：**

1. **库位热力图**
   - 系统记录每个库位的历史访问频次
   - 生成"热力图"，标注高频库位
   - 路径规划时优先经过低频库位，减少拥堵

2. **批量任务合并**
   - 系统将多个补货任务合并为一个"补货批次"
   - 叉车工一次领取多个任务
   - 系统计算最优路径（TSP旅行商问题算法）

3. **动态路径调整**
   - 执行过程中，如遇到障碍物或拥堵
   - Handheld端实时重新计算路径

**示例：**

```
传统路径（未优化）：
任务1：6F-A-01 → 3F-A-01
任务2：6F-B-05 → 3F-B-03
任务3：6F-A-10 → 3F-A-05
路径：6F-A-01 → 3F-A-01 → 6F-B-05 → 3F-B-03 → 6F-A-10 → 3F-A-05
总路程：1000米

优化后路径：
合并任务：一次领取3个任务
路径：6F-A-01（取货）→ 6F-A-10（取货）→ 6F-B-05（取货）
     → 3F-A-01（放货）→ 3F-A-05（放货）→ 3F-B-03（放货）
总路程：600米（节省40%）
```

## 6. 异常处理

### 6.1 异常类型与处理方案

| 异常类型 | 触发条件 | 处理方案 |
|---|---|---|
| **源库位无货** | 到存储区取货时，发现库存为0或不足 | 1. 标记任务为"异常"<br>2. 通知仓库经理<br>3. 人工介入处理（如：从其他库位调拨） |
| **目标库位已满** | 到拣货区放货时，发现库位容量不足 | 1. 系统自动查找附近空库位<br>2. 或：通知仓库经理手动指定新库位 |
| **SKU不匹配** | 扫描SKU条码时，与任务不符 | 1. Handheld提示错误<br>2. 禁止继续操作<br>3. 需重新扫描正确SKU |
| **库位不匹配** | 扫描库位条码时，与任务不符 | 1. Handheld提示错误<br>2. 禁止继续操作<br>3. 需重新扫描正确库位 |
| **补货任务冲突** | 同一库位同时有多个补货任务 | 1. 系统自动合并任务<br>2. 或：提示用户选择保留哪个任务 |
| **补货取消** | 叉车工主动取消任务 | 1. 任务状态变为"已取消"<br>2. 记录取消原因<br>3. 任务可重新分配 |
| **中转容器丢失** | 补货过程中容器条码无法扫描 | 1. 支持手动输入容器编号<br>2. 或：重新绑定新容器 |

### 6.2 异常处置流程

```
[发现异常] → [Handheld端上报异常] → [系统记录异常日志]
    ↓
[仓库经理接收通知] → [查看异常详情] → [决策：自动处理 or 人工处理]
    ↓
[自动处理]：系统自动重新分配任务、查找替代库位等
[人工处理]：仓库经理手动介入，如调整任务、重新盘点等
    ↓
[异常处理完成] → [记录处理结果] → [通知相关人员]
```

## 7. 主动补货功能

### 7.1 功能说明

除了系统自动触发的补货任务外，还应支持**主动补货**功能：

- 仓库人员在PC端或Handheld端手动创建补货任务
- 适用于：紧急补货、特殊处理、系统未触发的场景

### 7.2 主动补货流程

```
操作流程（PC端）：
1. 仓储管理 → 补货管理 → 创建补货任务
2. 填写补货信息：
   - SKU（搜索或扫描条码）
   - 补货数量
   - 来源库位（系统自动推荐或手动选择）
   - 目标库位（系统自动推荐或手动选择）
   - 优先级（高/中/低）
   - 备注
3. 点击"保存" → 任务创建成功
4. 任务进入"待领取"状态

操作流程（Handheld端）：
1. 登录 → 选择"补货管理" → "创建补货任务"
2. 扫描SKU条码
3. 系统自动显示：当前库存、推荐来源库位、推荐目标库位
4. 输入补货数量
5. 点击"提交" → 任务创建成功
```

### 7.3 主动补货与自动补货的协调

**问题：** 如果同时存在自动补货任务和主动补货任务，如何处理？

**解决方案：**

1. **去重逻辑**
   - 创建主动补货任务时，系统检查是否已有相同SKU和库位的自动补货任务
   - 如有，提示用户："已存在自动补货任务，是否继续创建主动补货任务？"
   - 用户确认后，系统自动取消或合并任务

2. **优先级管理**
   - 主动补货任务可设置更高优先级
   - 系统优先分配高优先级任务

## 9. 数据分析与优化

### 9.1 补货效率分析

**指标：**

| 指标 | 说明 | 计算公式 |
|---|---|---|
| **补货任务数** | 一定时间内的补货任务总数 | 计数 |
| **补货完成率** | 成功完成的补货任务占比 | 已完成任务数 / 总任务数 |
| **平均补货时长** | 从任务生成到完成的平均时间 | 总时长 / 完成任务数 |
| **补货及时率** | 在要求时间内完成的补货任务占比 | 及时完成任务数 / 总任务数 |
| **补货距离** | 平均补货距离 | 总距离 / 完成任务数 |

**报表：**

- 补货效率日报/周报/月报
- 按SKU、库位、人员的补货效率分析
- 补货异常统计

### 9.2 库位优化分析

**问题：** 哪些库位需要优化？

**分析方法：**

1. **补货频次分析**
   - 统计每个库位的补货频次
   - 频次过高的库位 → 容量不足 → 建议扩大容量或调整商品
   - 频次过低的库位 → 容量过剩 → 建议缩小容量或调整商品

2. **库位容量利用率分析**
   - 统计每个库位的平均库存占用率
   - 利用率过高的库位 → 扩容或调整商品
   - 利用率过低的库位 → 缩容或调整商品

3. **库位距离分析**
   - 统计存储区到拣货区的平均距离
   - 距离过远的库位 → 调整商品存放位置

**优化建议输出：**

```
系统自动生成优化建议：
1. SKU-001（矿泉水）在3F-A-01-03库位的补货频次为50次/天，
   建议：扩大该库位容量从100件到200件，或将该SKU调整到更大的库位。
2. SKU-002（毛巾）在6F-B-05-02库位的补货频次为2次/月，
   建议：缩小该库位容量，或将该SKU调整到更小的库位。
3. 6F-C区到3F-A区的平均补货距离为80米，
   建议：将高频补货的SKU调整到更近的存储区（如6F-A区）。
```

### 9.3 补货策略优化

**A/B测试：**

- 对同一SKU，分别使用不同的补货模型
- 对比补货效率、库存周转率、缺货率等指标
- 选择最优模型

**动态调整：**

- 系统根据历史数据，动态调整补货参数
- 如：根据季节性需求波动，自动调整安全库存

## 10. 与其他模块的集成

### 10.1 与拣货模块的集成

**场景：** 拣货时发现拣货区库存不足

**集成逻辑：**

```
[拣货任务生成] → [检查拣货区库存]
    ↓ 库存不足
[系统自动生成补货任务] → [关联拣货任务ID]
    ↓
[补货完成] → [通知拣货员] → [继续拣货]
```

**关键设计：**

- 补货任务与拣货任务关联（`pick_task_id`字段）
- 补货完成后，系统自动通知拣货员
- 拣货员可在Handheld端查看补货进度

## 11. 实施建议

### 11.1 分阶段实施

| 阶段 | 功能范围 | 说明 |
|---|---|---|
| **Phase 1** | 基础补货功能 | 支持(s,S)模型、手工创建补货任务、PC端执行 |
| **Phase 2** | Handheld端补货 | 支持Handheld端领取、执行、确认补货任务 |
| **Phase 3** | 自动补货触发 | 支持库存阈值触发、定时触发 |
| **Phase 4** | 高级功能 | 支持多种补货模型、路径优化、动态预警 |
| **Phase 5** | 数据分析 | 补货效率分析、库位优化建议 |

### 11.2 配置建议

**初期配置（简化版）：**

- 补货模型：仅支持(s,S)模型
- 触发判断依据：账面库存
- 检查周期：10分钟
- 任务分配策略：抢单模式（简单场景）

**成熟期配置（完整版）：**

- 补货模型：支持全部四种模型
- 触发判断依据：账面库存
- 检查周期：5分钟
- 任务分配策略：自动分配（优化效率）
- 路径优化：启用
- 动态预警值：启用（高级功能）
