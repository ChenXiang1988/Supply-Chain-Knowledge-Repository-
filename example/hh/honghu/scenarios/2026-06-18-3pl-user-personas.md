# 3PL 平台用户画像清单

文档类型：用户画像  
版本：V0.1  
日期：2026-06-18  
作者：Martin  
相关方：产品、架构、研发、仓储运营、关务、财务、客户成功

## 1. 目标

这份文档用来回答一个最现实的问题：这套 3PL 平台到底给谁用。

如果用户画像不清楚，后续的场景、页面、权限、数据口径都会散掉。

## 2. 用户分层

平台用户可以分成七层：

1. 内部总部与区域管理
2. 客户成功与服务
3. 仓库现场管理
4. 仓库现场执行（入库 / 出库 / 库存）
5. 客户 / 货主
6. 消费者（货主的 C 端客户）
7. 海外合作方 / 外部协同方

### 3.4 仓库现场执行

仓库执行端按作业链路拆分为 8 个角色。单个仓库规模从 10 人到 600 人不等，角色之间必须职责分明，每个人在 Handheld 上只看到自己该做的任务。

#### 3.4.1 入库团队

| 角色 | 主要目标 | 高频动作 | 看到什么 | 不该看到什么 | 终端 | 首批重要性 |
|---|---|---|---|---|---|---|
| 收货员 | 快速准确完成到货登记和收货确认 | 扫描入库订单号、清点箱数/托盘数、扫描条码或录入 SKU 编码确认收货 | 到货登记页、收货确认页、当前入库订单货物清单 | 拣货任务、打包任务、库位推荐 | Handheld | 高 |
| 验货员 | 核对货物与 ASN 是否一致，发现并登记差异 | 逐箱扫描条码/SKU码、对比预报数量、登记少收/多收/破损/错货、拍照存证 | 验货清单（按入库订单）、差异登记入口、拍照功能 | 拣货任务、库位管理、发运 | Handheld | 高 |
| 上架员 | 把验货合格的货物放入正确库位 | 扫描 SKU、查看推荐库位、扫描库位确认、批量上架（整托） | 待上架清单、推荐库位、库位确认页 | 订单信息、拣货路径、复核 | Handheld | 高 |

|---|---|---|---|---|---|---|

### 3.6 消费者（退货场景）

消费者是货主的 C 端客户，不直接使用 3PL 平台。在退货（S-08）场景中，消费者向货主发起退货申请，货主审核后通过 3PL 平台生成退货单。消费者与 3PL 仓库之间没有系统交互——3PL 只对货主负责。

| 角色 | 与平台的关系 | 涉及场景 | 首批重要性 |
|---|---|---|---|
| 消费者 | 通过货主间接关联，无 3PL 平台账号 | S-08 退货（发起退货申请→寄回商品→收货退款/换货） | P1（S-08 首批不纳入） |

### 3.9 仓库执行端管理关系

```
仓库经理
  ├── 班组长（入库）        ← 管：收货员、验货员、上架员
  ├── 班组长（出库）        ← 管：拣货员、复核员、打包员、发运员
  ├── 班组长（库存 + 设备）  ← 管：库存管理员、叉车工
  └── 班组长（夜班/轮班）    ← 同上，不同班次
```

每个角色在 Handheld 上只看到自己职责范围内的任务页面。班长组长可以看到本组所有任务的状态总览和每个组员的完成进度。

## 4. 首批优先用户

如果只选一批用户做试点，建议优先考虑：

1. 仓库经理
2. 入库班组（收货员、验货员、上架员）
3. 出库班组（拣货员、复核员、打包员、发运员）
4. 客户管理员
5. 总部运营负责人
6. 客户成功

原因很简单：

- 入库和出库班组覆盖最核心的执行闭环
- 客户管理员是 SKU 注册和贴码管理的责任人
- 仓库经理和班组长暴露现场管理和异常处置问题
- 总部运营和客户成功验证规则配置和客户服务链路
- 这些角色能最快验证系统是否真的可用

## 6. 待确认项

| 待确认项 | 为什么重要 |
|---|---|
| 首批试点是国内仓、海外仓还是保税仓 | 决定场景优先级和合规复杂度 |
| 首批客户是 B2B、B2C 还是混合 | 决定页面和履约方式 |
| 首批是否需要客户门户 | 决定是否把协同层纳入第一版 |
| 首批是否需要合作方回传 | 决定集成和权限设计 |
