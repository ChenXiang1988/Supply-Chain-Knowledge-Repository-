前面我们做了用户系统设计实战。

这一讲，我们继续看另一个数据库设计中的经典场景：

订单表设计。

如果说用户表几乎所有系统都会有，那订单表就是电商、支付、交易类系统里最核心的表之一。

订单表设计得好不好，会直接影响后面的查询、统计、支付流程、售后流程，甚至影响整个系统的可扩展性。

这一讲，我们就来看看订单表应该怎么设计。

---

一、订单表到底要存什么

设计订单表之前，先不要急着写字段。

先想清楚：

一张订单，业务上到底包含哪些关键信息？

通常至少包括：

* 订单 id
* 订单编号
* 用户 id
* 订单金额
* 订单状态
* 创建时间

如果业务再完整一些，还可能包括：

* 支付时间
* 收货地址
* 备注信息
* 支付方式
* 更新时间

所以订单表设计的核心，就是把“订单这个业务对象”的关键状态存完整。

---

二、一张基础订单表可以怎么设计

例如：

```
CREATE TABLE orders (  
  id INT PRIMARY KEY AUTO_INCREMENT,  
  order_no VARCHAR(50) NOT NULL UNIQUE,  
  user_id INT NOT NULL,  
  total_amount DECIMAL(10,2) NOT NULL,  
  status VARCHAR(20) NOT NULL,  
  pay_type VARCHAR(20),  
  remark VARCHAR(255),  
  create_time DATETIME,  
  pay_time DATETIME,  
  update_time DATETIME  
);
```

这里：

* `id` 是主键
* `order_no` 是订单编号，通常要求唯一
* `user_id` 表示这张订单属于哪个用户
* `total_amount` 表示订单总金额
* `status` 表示订单当前状态
* `pay_type` 表示支付方式
* `remark` 表示备注
* 各种时间字段用来记录订单生命周期

---

三、为什么订单编号通常要单独设计

很多初学者会问：

既然已经有 `id` 了，为什么还要再加一个 `order_no`？

因为：

* `id` 更像数据库内部主键
* `order_no` 更像业务侧订单号

在真实系统里，用户看到的、客服查询的、支付系统使用的，往往是订单编号，而不是数据库自增 id。

所以订单号通常需要单独设计，而且一般要求唯一。

---

四、为什么状态字段特别重要

订单不是静态数据，它会不断变化。

例如：

* 待支付
* 已支付
* 已发货
* 已完成
* 已取消

所以订单表里，状态字段几乎是必不可少的。

很多后续业务逻辑，都是围绕订单状态展开的。

比如：

* 待支付才能付款
* 已支付才能发货
* 已完成后才能评价

所以状态字段不仅是展示用途，也是业务流程控制的一部分。

---

五、订单和用户是什么关系

订单和用户通常是：

一对多关系。

也就是说：

* 一个用户可以有多个订单
* 一张订单只属于一个用户

所以订单表里放 `user_id` 是很自然的设计。

---

六、订单表和订单明细表为什么经常分开

很多初学者一开始会把订单和商品信息全放在同一张表里。

但真实设计中，通常会拆成：

* 订单主表
* 订单明细表

例如：

```
CREATE TABLE order_items (  
  id INT PRIMARY KEY AUTO_INCREMENT,  
  order_id INT NOT NULL,  
  product_id INT NOT NULL,  
  price DECIMAL(10,2),  
  quantity INT  
);
```

这样做的原因是：

一张订单里可能有多个商品。

所以：

* 订单主表存“订单整体信息”
* 明细表存“每个商品项的信息”

这本质上就是一对多关系。

---

七、时间字段为什么要尽量补齐

订单系统里，时间字段非常重要。

例如：

* `create_time` 记录下单时间
* `pay_time` 记录支付时间
* `update_time` 记录最后更新时间

这些字段后面会影响很多操作：

* 查询最近订单
* 统计今天成交额
* 判断超时未支付
* 做售后和对账

所以订单类表通常比普通表更依赖时间字段。

---

八、初学者常见错误

1）订单表里没有订单编号，只有自增 id

2）订单状态设计得太随意，后期流程难控制

3）一张订单多个商品，结果硬塞在一个字段里

4）金额字段类型选得不合理

5）缺少关键时间字段，后期统计和排查都很麻烦

所以订单表设计的重点，不是字段写得多，而是要把业务流程中的关键节点表达清楚。

九、小总结

这一讲你需要掌握：

1）订单表是交易系统中的核心表之一

2）订单表通常要包含订单编号、用户 id、金额、状态、时间等关键字段

3）订单编号通常要单独设计，并保证唯一

4）订单和用户通常是一对多关系

5）订单主表和订单明细表通常也要拆开设计

6）状态字段和时间字段在订单系统里非常重要

7）订单表设计要服务后续流程、查询和统计
